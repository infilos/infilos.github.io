Span

整个追踪树是由多个带有关联关系的 Span 构成，Span 也作为追踪信息的最小单元。

Span

如上图，SpanCustomier 是主接口，定义了 OpenTracing 规范中的基本操作，包括名称、标签、标注。如果业务中需要自定义一些字段以便对 Span 进行查询，可以基于该接口进行扩展。

Span 是一个抽象类，建模了 Span 的所有操作，甚至包括 Span 的生命周期管理，暴露给用户的 Span 也均为该类型。其内部实现了一个枚举 Kind 用于表示一个 Span 的位置角色，包括：CLIENT、SERVER、PRODUCER、CONSUMER。

ScopedSpan 同样是一个抽象类，根据注释，”在单进程追踪过程中，可以被认为是一个简化 API 后的 Span“。

对应着三种基本抽象，Brave 提供了两种实现 “Noop-” 和 “Real-”，“Noop-” 意味着不做任何操作，当用户没有进行正确的配置时、或者当前追踪不需要被采样时会返回该类 Span 实例；“Real-” 则是对各类抽象的实现，并且封装了 Span 的可变 API。

此外，还提供了一个 CurrentSpanCustomizer，用于为外部提供一个不为 Null 的 Span 实例，以避免用户进行必要的非 Null 检查。

在 RealSpan 中我们可以看到 Span 的具体操作实现，其实现主要依赖一下核心类：

  • Recorder：用于在本地记录 Span。
  • Reporter：用于将 Span 上报给 Zipkin。
  • MutableSpan：Span 的(可变)包装类，提供各种 API 操作 Span。
  • MutableSpanMap：以 TraceContext 为键，MutableSpan 为值的 Map 结构，用于在内存中保存所有活动的 Span。

Start/Finish

@Override public Span start(long timestamp) {
    recorder().start(context());
    return this;
}
  
@Override public void finish(long timestamp) {
	recorder().finish(context(), timestamp);
}

这两个方法内部分别会去调用 Recorder 的 start 和 finish 方法，而在 Recorder 内部，这两个方法实际会去调用用于保存所有本地 Span 的 MutableSpanMap 的对应方法，并调用 Reporter 来发送 Span 数据。

需要注意的是，应尽量调用这两个方法的无参重载方法，交由系统来完成对时间戳、持续时长的计算，以避免计算错误。

其他操作

  • name()
  • kind()
  • annotate()
  • tag()
  • error()
  • remoteEndpoint()
  • abandon()
  • flush()

所有这些方法都是基于对 Recorder 的操作，来实现对 Span 信息的创建、更新、发送。在 Recorder 部分我们将详细展开。
