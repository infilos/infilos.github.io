OpenTracing

数据模型

OpenTracing 中的 Trace 由其 Span 隐式定义。**Trace** 可以被认为是由一组 Span 定义的有向无环图(DAG)，**Span** 之间的关系被称为 Reference。

以下是一个由 8 个 Span 构成的 Trace 的结构：

一个 Trace 中 Span 之间的因果关系：

        [Span A]  ←←←(the root span)
            |
     +------+------+
     |             |
 [Span B]      [Span C] ←←←(Span C之于Span A的关系是 `ChildOf` )
     |             |
 [Span D]      +---+-------+
               |           |
           [Span E]    [Span F] >>> [Span G] >>> [Span H]
                                       ↑
                                       ↑
                         (Span G `FollowsFrom` Span F)

有时使用时间轴来显示 Trace 则更为直观：

一个 Trace 中 Span 之间的时间关系

––|–––––––|–––––––|–––––––|–––––––|–––––––|–––––––|–––––––|–> time

 [Span A···················································]
   [Span B··············································]
      [Span D··········································]
    [Span C········································]
         [Span E·······]        [Span F··] [Span G··] [Span H··]

每个 Span 中封装了如下信息：

  • 操作名称；
  • 开始时间戳；
  • 结束时间戳；
  • 一组包含零个或多个*键值*结构的 Span 标签(Tags)。键必须是字符串，值可以是字符串、布尔值、数字；
  • 一组包含零个或多个 Span 日志(Logs)，每条日志都是一个键值映射，并与一个时间戳配对。键必须是字符串，值可以是任何类型(但并非所有的 OpenTracing 实现都必须支持每种值类型)；
  • 一个 SpanContext 上下文；
  • 零个或多个因果相关的 Span 间的 Reference，基于对应 Span 的 SpanContext 进行关联。

每个 SpanContext 封装了如下状态：

  • 任何需要跟跨进程 Span 关联的、依赖于 OpenTracing 实现的状态，比如 Trace 和 Span 的 ID；
  • 键值结构的、需要跨进程传递的 Baggage Items(随身行李条目)。

Span 间的 Reference

一个 Span 可以引用零个或多个因果相关的 SpanContext。OpenTracing 目前定义类两种类型的 Reference：ChildOf 和 FollowsFrom。两种 Reference 类型都对父子 Span 间的关系进行了建模。未来，OpenTracing 可能会为不具因果关系的 Span 提供不同类型的 Reference(比如一组批量 Span、阻塞在队列中的 Span 等)。

ChildOf Reference

一个 Span 作为另一个 Span 的子 Span。在 ChildOf 引用中，父 Span 在某种程度上取决于子 Span。下列情况会构成 ChildOf 关系：

  • 在一个 RPC 中，代表服务端的 Span 可以使用 ChildOf 关联到代表客户端的父 Span。
  • 在一次持久化过程中，代表 SQL 插入的 Span 可以使用 ChildOf 关联到代表 ORM Save 方法的父 Span。
  • 多个并发(可能是分布式)执行任务的 Span 可能分别使用 ChildOf 关联到合并了所有子 Span 执行结果的父 Span。

下列都是具有有效的 ChildOf 关系的时序图：

[-Parent Span---------]
	[-Child Span----]

[-Parent Span--------------]
	[-Child Span A----]
	[-Child Span B----]
	[-Child Span C----]
	[-Child Span D---------------]
	[-Child Span E----]

FollowsFrom Reference

有些父 Span 不依赖于任何子 Span 的结果。这种情况下，我们仅认为子 Span 在因果上 FollowsFrom 父 Span。有多种不同的 FollowsFrom 引用子类别，在 OpenTracing 的未来版本中，它们可能会被更正式的区分。

下列都是具有有效的 FollowsFrom 关系的时序图：

[-Parent Span-]  [-Child Span-]

[-Parent Span--]
	[-Child Span-]

[-Parent Span-]
	[-Child Span-]

接口定义

OpenTracing 规范中有三种相互关联的关键类型 Tracer、Span、SpanContext。下面我们将介绍各种类型的行为；简单来说，每种行为在编程语言中分别对应一个“方法”，尽管实际上可能是一组重载方法。

当我们讨论可选参数时，应当清楚，不同的语言有不同的方式来解释这个概念。比如 Go 语言中我们会使用 “Functional Options” 这个术语，而在 Java 中我们会使用 Builder 模式。

Tracer

Tracer 接口能够创建 Span 并且能够跨进程地 Inject(序列化) 和 Extract(反序列化)。严格来说，它应该具备以下能力：

启动一个新 Span

必要参数：

  • 操作名称，一个人类可读的、富有意义的、简练字符串，它简洁地表示由 Span 完成的工作，比如：RPC 方法名称、函数名称、一个较大计算任务的其中一个阶段的名称。操作名称应用用**泛化**的字符串形式来标识一个 Span 实例，也就是说，get_user 比 get_user/12345 要好。

可选参数：

  • 零个或多个与 SpanContext 相关的 Reference，尽可能包括 ChildOf 和 FollowsFrom 引用类型的信息。
  • 开始时间戳，如果没有的话，默认使用现实时间(walltime)。
  • 零个或多个标签(tag)

返回值是一个刚启动的 Span 实例。

注入(Inject) SpanContext 到载体中

必要参数：

  • SpanContext 实例。
  • 格式描述符(format descriptor)(通常但不一定是字符串常量)，告诉 Tracer 实现如何在载体对象中对 SpanContext 进行编码。
  • 载体(carrier)，其类型由类型描述符指定。Tracer 的实现将根据**格式描述符**对此载体对象中的 SpanContext 进行编码。

从载体中提取(Extract) SpanContext

必要参数：

  • 格式描述符，告诉 Tracer 实现如何在载体对象中对 SpanContext 进行解码
  • 载体，其类型由类型描述符指定。Tracer 的实现将根据**格式描述符**对此载体对象中的 SpanContext 进行解码。

返回值是一个 SpanContext 实例，适合作为通过 Tracer 启动 Span 时的 reference。

注意：Injection/Extraction 所需的格式

Injection 和 Extraction 都依赖于一个可扩展的格式描述符参数，它指定了相关联的“载体”类型以及 SpanContext 是如何被编码的。Tracer 实现必须支持下列所有格式。

  • 文本映射(Text Map)：任意的无字符编码限制的 String-String 型键值结构。
  • HTTP Headers：String-String 型键值结构在 HTTP Headers(RFC 7230)中也适用。在实践中，由于 HTTP Headers 容易被各种不同的方式处理，强烈建议在 Tracer 实现中使用有限的键空间和保守的转义值。
  • 二进制：表示了 SpanContext 的任意二进制数据。

Span

除了获取 Span 的 SpanContext 的方法之外，下列任何方法都不能在 Span 结束后被调用。

获取 Span 的 SpanContext

无需参数。返回值是 Span 的 SpanContext。返回值甚至能够在 Span 结束后被使用。

重写操作名

必要参数：

  • 新的**操作名**，当 Span 启动时取代旧的操作名。

结束 Span

可选参数：

  • 显式添加 Span 的结束时间戳，如果不提供该参数则会默认使用现实时间。

设置 Span 标签

必要参数：

  • 标签的键，必须是字符串。
  • 标签的值，必须是字符串、布尔值、数值。

注意 OpenTracing 中已经为一些**标准标签**规定了语义。

记录(Log)结构化的数据

必要参数：

  • 一个或多个键值对，键必须是字符串，值可以是任意类型。有些 OpenTracing 的实现可以处理更多的日志值。

可选参数：

  • 显式时间戳。该参数必须处于当前 Span 的开始与结束时间戳之间。

注意 OpenTracing 中已经为一些**日志标准键**规定了语义。

设置 Baggage Item

Baggage Item 是对应于某个 Span 及其 SpanContext，以及**所有直接或间接地引用自本地(local) Span 的 Span 的键值对**。也就是说，Baggage Item 是随 Trace 一起传播的。

Baggage Item 为全栈式的 OpenTracing 集成提供了强大的功能(比如在移动 APP 上使用时，它可以一路追踪数据直至后端存储系统的深度)，不过使用这些功能时要当心。

每个键值都会被拷贝到每个本地或远程的子 Span，这可能导致巨大的网络或 CPU(序列化/反序列化) 开销。

必要参数：

  • Baggage 键，字符串
  • Baggage 值，字符串

获取 Baggage Item

必要参数：

  • Baggage 键，字符串

SpanContext

在一般所指的 OpenTracing 层面上，SpanContext 更像是一个概念而非一种具体的功能。也就是说，这对 OpenTracing 的实现来说提供一层薄的接口是有必要的。当启动一个 Span 或注入、提取 Trace 时，多数 OpenTracing 用户仅需通过 Reference 与 SpanContext 进行交互。

在 OpenTracing 中，SpanContext 被强制设定为不可变的(immutable)，以应对在 Span 结束时和被引用时产生的复杂生命周期的问题。

遍历所有的 Baggage Item

不同编程语言对此有不同的建模方式，不过给出一个 SpanContext 实例，语义上还是能让该调用者在短时间内遍历所有关联的 Baggage Item。

NoopTracer

所有编程语言的 OpenTracing API 必须提供某种 NoopTracer(no-op，无操作) 实现，可以用作标记控制(flag-control)或者进行一些用于测试的无害注入等。某些情况下(比如 Java)，NoopTracer 可能在它自己的制品包(packaging artifact)中。

可选的 API 元素

有些语言提供了在单进程中用来传递活动的 Span 和 SpanContext 工具。比如，opentracing-go 提供的 Go 语言 context.Context 机制中，用 set 和 get 来获得或设置 Span 的帮助函数。
