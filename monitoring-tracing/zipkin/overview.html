Overview

Zipkin是一款开源的分布式实时数据追踪系统(Distributed Tracking System)，基于 Google Dapper 的论文设计而来，由 Twitter 公司开发贡献。其主要功能是聚集来自各个异构系统的实时监控数据。

架构概览

应用中的追踪器(Tracer)将会记录所发生的操作的时间(Timing)及元信息(Metadata)。这些追踪器通常是检测仪器类库，因此对于用户来说通常是透明的。比如，一个被装备了检测仪器类库的 Web 服务在接收到请求和返回响应时将会执行记录动作。被收集到的追踪数据则被称为一个 Span。

检测仪器类库会以生产安全的方式被编写，并拥有很小的负载。因此它们仅在带内(in-band)传播一些 ID，以便告诉接收器正在进行的追踪。完成的 Span 将会被发送给带外(out-of-band) Zipkin，类似于应用异步的发送度量信息的方式。

比如，当一个需要对外发起 HTTP 请求的操作被追踪，几个请求头会被添加到传播 ID。而请求头并不会携带其他细节，比如被追踪的操作名称。

在被装置监控仪器的(Instrumented)应用中，向 Zipkin 发送数据的组件被称为 Reporter。Reporter 使用多种传输器中的一种将追踪数据发送给 Zipkin 的接收器，而接收器会将数据持久化到存储中。然后，API 会查询存储中的数据以在 UI 中展示。

如上图，应用代码发起 Http Get 请求(请求路径/foo)，经过 Trace 框架拦截，并依次经过如下步骤，记录Trace 信息到 Zipkin 中：

  1. 记录 tags 信息
  2. 将当前调用链的 Trace 信息记录到 Http Headers 中
  3. 记录当前调用的时间戳(timestamp)
  4. 发送 HTTP 请求，并携带 Trace 相关的 Header，如 X-B3-TraceId:aa，X-B3-SpandId:6b
  5. 调用结束后，记录当次调用所花的时间(duration)
  6. 将步骤 1-5，汇总成一个Span(最小的Trace单元)，异步上报该 Span 信息给 Zipkin Collector

数据模型

一个 Span 的示例数据：

{
    "traceId": "string",
    "name": "string",
    "parentId": "string",
    "id": "string",
    "kind": "CLIENT",
    "timestamp": 0,
    "duration": 0,
    "debug": true,
    "shared": true,
    "localEndpoint": {
      "serviceName": "string",
      "ipv4": "string",
      "ipv6": "string",
      "port": 0
    },
    "remoteEndpoint": {
      "serviceName": "string",
      "ipv4": "string",
      "ipv6": "string",
      "port": 0
    },
    "annotations": [
      {
        "timestamp": 0,
        "value": "string"
      }
    ],
    "tags": {
      "additionalProp1": "string",
      "additionalProp2": "string",
      "additionalProp3": "string"
    }
  }

更多数据模型可以查看 Zipkin 的 API 文档。

数据传输

被追踪器发送的 Span 必须在被追踪的服务上发送给 Zipkin 的收集器。有三种主要的传输器：HTTP、Kafka、Scribe。

基本组件

Zipkin Collector

数据一旦抵达 Zipkin 的收集器程序，则会被收集器验证、存储、索引以便查询。

Storage

Zipkin 最初构建于将数据存储到 Cassandra，因为其可扩展、拥有灵活的 Schema，并且在 Twitter 内大量使用。然而，我们将该组件实现为可插拔的形式。目前原生支持 ElasticSearch 和 MySQL，并能通过三方扩展实现对其他存储的支持。

Zipkin Query Service

一旦数据被存储并索引，我们需要一种方式来检索它们。查询程序提供了一种简单的 JSON API 用于查找或检索追踪数据。该 API 的主要消费者则是用于展示追踪数据的 Web UI。

Web UI

Web UI 提供了一种直观的方式来基于服务、时间、注解来展示追踪数据。
